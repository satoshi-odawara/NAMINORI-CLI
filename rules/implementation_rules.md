# データ分析スクリプト実装規約（実装・運用ガイドライン）

本ドキュメントは、`gemini.md` 技術憲章に基づき、具体的なコーディング、解析アルゴリズム、およびツールの運用ルールを定義する。

---

## 1. 技術スタック

* **Language:** Python 3.10+
* **Frontend:** Streamlit
* **Data Processing:** NumPy, SciPy (signal, fft)
* **Visualization:** Plotly
* **Testing:** Pytest

---

## 2. 共通コーディング規約

### 2.1 物理量・単位の明示

* すべての数値は **物理量 + 単位** を前提とする。
* 変数名には必ず単位を含める。
  * 例：`accel_ms2`, `vel_mms`, `disp_um`, `freq_hz`, `time_s`

### 2.2 物理量種別の明示（必須）

加速度・速度・変位を混在させないため、解析関数には必ず物理量種別を明示する。

```python
from enum import Enum

class SignalQuantity(Enum):
    ACCEL = "accel"        # m/s^2
    VELOCITY = "velocity" # mm/s
    DISPLACEMENT = "disp" # μm
```

### 2.3 文字エンコーディングの取り扱い（新規）

* 日本語を含むデータの抽出や確認を行う際は、標準出力（print）に頼らず、必ず **UTF-8 エンコーディングで一時ファイルに書き出し**、`read_file` ツールで内容を確認すること。
* Windows環境下での文字化けを防止し、確実なデータ理解を行うための必須手順とする。

### 2.4 ドメイン知識の定数化（新規）

* チーム名、ダービー定義、天候カテゴリなどの「ドメイン知識に基づく定義」は、`scripts/constants.py` などの定数ファイルに集約すること。
* ロジックと知識を分離し、分析全体での整合性とメンテナンス性を担保する。

---

## 3. 運用・保守ガイドライン

### 3.1 `replace` ツール使用規約

* **厳密な一致:** `old_string` は改行・空白含めターゲットと完全に一致させること。
* **大規模変更:** リファクタリング時は `read_file` -> `write_file` 方式を推奨。

### 3.2 自動テストの資産化（新規）

* `./scripts/` に保存されるすべての主要な処理（前処理、モデル推論等）には、必ず `./tests/` に対応するテストコード（pytest形式）を作成・保存し、コミット対象に含めること。
