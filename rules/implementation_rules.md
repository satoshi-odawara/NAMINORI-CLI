# データ分析スクリプト実装規約（実装・運用ガイドライン）

本ドキュメントは、`gemini.md` 技術憲章に基づき、具体的なコーディング、解析アルゴリズム、およびツールの運用ルールを定義する。

---

## 1. 技術スタック

* **Language:** Python 3.10+
* **Frontend:** Streamlit
* **Data Processing:** NumPy, SciPy (signal, fft), Pandas
* **Visualization:** Plotly, Matplotlib, Seaborn
* **Testing:** Pytest

---

## 2. 共通コーディング規約

### 2.1 物理量・単位の明示

* すべての数値は **物理量 + 単位** を前提とする。
* 変数名には必ず単位を含める（例：`accel_ms2`, `count_num`, `time_s`）。

### 2.2 実行コンテキストの固定（必須）

* すべてのスクリプトは、**プロジェクトルートを作業ディレクトリ（CWD）として実行される**ことを前提に実装すること。
* データ読み込みや出力のパスは、ルートからの相対パス（例：`inputdata/train.csv`, `analysis/res_01_...`）で記述し、環境依存を排除する。

### 2.3 文字エンコーディングの取り扱い

* 日本語を含むデータの抽出や確認を行う際は、標準出力に頼らず、必ず **UTF-8 エンコーディングで一時ファイルに書き出し**、`read_file` ツール等で内容を確認すること。

### 2.4 ドメイン知識の分離

* 案件固有の定義（カテゴリ化基準、閾値、フラグ定義等）は、汎用ロジックから分離し、特定の定数ファイル（例：`constants.py`）または案件固有ルール（`PROJECT_SPECIFIC_RULES.md`）に集約すること。

---

## 3. 可視化と環境耐性 (Environment-Robust Visualization)

サンドボックス環境やクラウド実行環境において、日本語フォント欠如による文字化けを回避するため、以下の作法を厳守すること。

### 3.1 ラベルの英語化
- チャートの軸ラベル、レジェンド、タイトルには原則として英語を使用する。
- カテゴリ変数に日本語が含まれる場合、可視化の直前でマッピング辞書等を用いて英語に変換すること。
  - 例: `df['stage'].map({'Ｊ１': 'J1', 'Ｊ２': 'J2'})`

### 3.2 フォント設定のフォールバック
- `matplotlib` 等の設定では、特定の日本語フォントを指定せず、汎用的な `sans-serif` を使用すること。
- `plt.rcParams['font.family'] = 'sans-serif'`

---

## 4. 運用・保守・QAガイドライン

### 4.1 仮説の事前定義（厳守）
- **データ接触前の定義**: [Lead DS] および [Domain Specialist] は、分析対象データの中身を確認する前に、業界標準、物理法則、ビジネスルールに基づく「期待される結果（仮説）」を `analysis/analysis_plan.txt` に明文化すること。
- **目的**: データ駆動（Inductive）な結果との「ズレ」を客観的に評価するため。

### 4.2 四視点独立調査と不一致分析
- 各視点の成果は `./analysis/view_[視点名]/` に隔離保存し、QA はその間の矛盾を「重要発見」として抽出する。
- 矛盾が発生した場合、その原因が「計算ミス」か「未知の物理現象」かを切り分ける「不一致分析」を必須とする。

### 4.3 代表サンプルの抽出（データの顔）
- レポートに掲載する「データの顔（代表例）」を選定する際は、主観を排し、以下のいずれかの基準で機械的に抽出すること。
  - **Medoid（メドイド）**: 各クラスの全サンプルの中で、他との距離の合計が最小となる「最も中心的な」実サンプル。
  - **Centroid（セントロイド）に近いサンプル**: クラス重心に最も近い実サンプル。
- 抽出ロジックは `utils/` に資産化し、再現性を担保すること。

### 4.4 `replace` ツール使用規約
- **厳密な一致:** `old_string` は改行・空白含めターゲットと完全に一致させること。
